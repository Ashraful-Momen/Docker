## 1. Introduction to Docker Swarm {#introduction}

Docker Swarm is Docker's native orchestration solution that turns a pool of Docker hosts into a single virtual Docker host. It provides clustering functionality for Docker containers with features like service discovery, load balancing, scaling, and rolling updates.

### Key Features
- **Native clustering** for Docker containers
- **Declarative service model** using desired state
- **Service discovery** and **load balancing**
- **Rolling updates** with rollback capabilities
- **Multi-host networking**
- **Built-in security** with TLS encryption

### Swarm vs Other Orchestrators
- **Kubernetes**: More complex but feature-rich
- **Docker Swarm**: Simpler, Docker-native, easier learning curve
- **Nomad**: HashiCorp's solution, supports multiple workload types

## 2. Prerequisites {#prerequisites}

### System Requirements
```bash
# Minimum requirements per node:
# - 2 GB RAM
# - 2 CPU cores
# - 20 GB disk space
# - Docker Engine 17.06.0+ installed
# - Network connectivity between nodes

# Supported OS:
# - Linux (Ubuntu, CentOS, RHEL, Debian)
# - Windows Server 2016+
# - macOS (development only)
```

### Network Requirements
```bash
# Required ports:
# TCP port 2377 - cluster management communications
# TCP and UDP port 7946 - communication among nodes
# UDP port 4789 - overlay network traffic

# Firewall configuration (Ubuntu/Debian):
sudo ufw allow 2377/tcp
sudo ufw allow 7946/tcp
sudo ufw allow 7946/udp
sudo ufw allow 4789/udp

# Firewall configuration (CentOS/RHEL):
sudo firewall-cmd --add-port=2377/tcp --permanent
sudo firewall-cmd --add-port=7946/tcp --permanent
sudo firewall-cmd --add-port=7946/udp --permanent
sudo firewall-cmd --add-port=4789/udp --permanent
sudo firewall-cmd --reload
```

## 3. Setting Up Docker Swarm {#setup}

### Initialize Swarm Cluster

```bash
# On the manager node:
docker swarm init --advertise-addr <MANAGER-IP>

# Example:
docker swarm init --advertise-addr 192.168.1.100

# Output will show join commands for workers and managers
# To get join commands later:
docker swarm join-token worker
docker swarm join-token manager
```

### Join Worker Nodes

```bash
# On worker nodes, run the command from swarm init output:
docker swarm join --token SWMTKN-1-xxxx <MANAGER-IP>:2377
```

### Join Additional Manager Nodes

```bash
# For high availability, add more managers:
docker swarm join --token SWMTKN-1-xxxx <MANAGER-IP>:2377
```

### Verify Cluster Status

```bash
# List all nodes:
docker node ls

# Get detailed node information:
docker node inspect <NODE-ID>

# Check swarm status:
docker system info | grep -A 10 "Swarm:"
```

## 4. Swarm Architecture {#architecture}

### Manager Nodes
- Maintain cluster state using Raft consensus algorithm
- Schedule services and assign tasks to worker nodes
- Handle API requests and CLI commands
- Recommended odd number (3, 5, 7) for fault tolerance

### Worker Nodes
- Execute tasks assigned by manager nodes
- Run application containers
- Report task status back to managers
- Can be promoted to managers if needed

### Services and Tasks
- **Service**: Definition of tasks to execute on nodes
- **Task**: Individual container running on a node
- **Replica**: Number of identical tasks in a service

## 5. Managing Nodes {#managing-nodes}

### Node Operations

```bash
# List all nodes:
docker node ls

# Inspect node details:
docker node inspect <NODE-NAME>

# Update node metadata:
docker node update --label-add environment=production node1

# Drain node (stop scheduling new tasks):
docker node update --availability drain node1

# Make node active again:
docker node update --availability active node1

# Remove node from swarm:
docker node rm <NODE-NAME>

# Force remove unresponsive node:
docker node rm --force <NODE-NAME>
```

### Node Labels and Constraints

```bash
# Add labels to nodes:
docker node update --label-add type=cpu-intensive node1
docker node update --label-add zone=us-east-1a node2

# Remove labels:
docker node update --label-rm type node1

# Use labels in service constraints:
docker service create \
  --constraint 'node.labels.type == cpu-intensive' \
  --name cpu-app nginx
```

### Promoting and Demoting Nodes

```bash
# Promote worker to manager:
docker node promote <NODE-NAME>

# Demote manager to worker:
docker node demote <NODE-NAME>
```

## 6. Services and Tasks {#services-tasks}

### Creating Services

```bash
# Basic service creation:
docker service create --name web-server nginx

# Service with replicas:
docker service create --name web-server --replicas 3 nginx

# Service with port mapping:
docker service create \
  --name web-server \
  --replicas 3 \
  --publish 8080:80 \
  nginx

# Service with resource constraints:
docker service create \
  --name web-server \
  --replicas 3 \
  --limit-cpu 0.5 \
  --limit-memory 512M \
  --reserve-cpu 0.25 \
  --reserve-memory 256M \
  nginx

# Service with environment variables:
docker service create \
  --name database \
  --env MYSQL_ROOT_PASSWORD=secretpassword \
  --env MYSQL_DATABASE=myapp \
  mysql:8.0
```

### Managing Services

```bash
# List services:
docker service ls

# Inspect service:
docker service inspect web-server

# View service logs:
docker service logs web-server

# Follow service logs:
docker service logs -f web-server

# Scale service:
docker service scale web-server=5

# Update service:
docker service update --image nginx:1.20 web-server

# Remove service:
docker service rm web-server
```

### Service Tasks

```bash
# List tasks for a service:
docker service ps web-server

# List all tasks:
docker service ps --no-trunc web-server

# View task logs:
docker service logs <TASK-ID>
```

### Rolling Updates

```bash
# Configure update behavior:
docker service create \
  --name web-server \
  --replicas 6 \
  --update-parallelism 2 \
  --update-delay 10s \
  --update-failure-action rollback \
  --update-max-failure-ratio 0.1 \
  nginx:1.19

# Perform rolling update:
docker service update --image nginx:1.20 web-server

# Rollback to previous version:
docker service rollback web-server
```

## 7. Docker Stack and Compose {#stack-compose}

### Docker Stack Overview
Docker Stack allows deploying multi-service applications using Docker Compose files in Swarm mode.

### Sample Stack File (docker-compose.yml)

```yaml
version: '3.8'

services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
    deploy:
      replicas: 3
      restart_policy:
        condition: on-failure
        delay: 5s
        max_attempts: 3
      placement:
        constraints:
          - node.role == worker
      resources:
        limits:
          cpus: '0.5'
          memory: 256M
        reservations:
          cpus: '0.25'
          memory: 128M
    networks:
      - frontend
    configs:
      - source: nginx_config
        target: /etc/nginx/nginx.conf
    secrets:
      - ssl_certificate

  api:
    image: myapp:latest
    deploy:
      replicas: 2
      update_config:
        parallelism: 1
        delay: 10s
        failure_action: rollback
        order: start-first
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/myapp
    networks:
      - frontend
      - backend
    depends_on:
      - db

  db:
    image: postgres:13
    deploy:
      replicas: 1
      placement:
        constraints:
          - node.labels.type == database
    environment:
      - POSTGRES_DB=myapp
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD_FILE=/run/secrets/db_password
    volumes:
      - db_data:/var/lib/postgresql/data
    networks:
      - backend
    secrets:
      - db_password

networks:
  frontend:
    driver: overlay
    attachable: true
  backend:
    driver: overlay
    internal: true

volumes:
  db_data:
    driver: local

configs:
  nginx_config:
    file: ./nginx.conf

secrets:
  ssl_certificate:
    file: ./ssl.crt
  db_password:
    file: ./db_password.txt
```

### Stack Operations

```bash
# Deploy stack:
docker stack deploy -c docker-compose.yml myapp

# List stacks:
docker stack ls

# List services in stack:
docker stack services myapp

# List tasks in stack:
docker stack ps myapp

# Remove stack:
docker stack rm myapp

# Update stack (redeploy with changes):
docker stack deploy -c docker-compose.yml myapp
```

### Advanced Stack Configuration

```yaml
# Production-ready stack configuration
version: '3.8'

services:
  web:
    image: nginx:alpine
    ports:
      - target: 80
        published: 80
        protocol: tcp
        mode: ingress
    deploy:
      mode: replicated
      replicas: 3
      endpoint_mode: vip
      restart_policy:
        condition: on-failure
        delay: 5s
        max_attempts: 3
        window: 120s
      update_config:
        parallelism: 1
        delay: 10s
        failure_action: rollback
        monitor: 60s
        max_failure_ratio: 0.1
        order: start-first
      rollback_config:
        parallelism: 1
        delay: 0s
        failure_action: pause
        monitor: 60s
        order: stop-first
      placement:
        constraints:
          - node.role == worker
        preferences:
          - spread: node.labels.zone
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:80"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 60s
```

## 8. Networking in Swarm {#networking}

### Overlay Networks

```bash
# Create overlay network:
docker network create --driver overlay my-network

# Create overlay network with custom subnet:
docker network create \
  --driver overlay \
  --subnet 10.0.1.0/24 \
  --gateway 10.0.1.1 \
  my-network

# Create encrypted overlay network:
docker network create --driver overlay --opt encrypted my-secure-network

# List networks:
docker network ls

# Inspect network:
docker network inspect my-network

# Remove network:
docker network rm my-network
```

### Service Networking

```bash
# Create service with custom network:
docker service create \
  --name web-server \
  --network my-network \
  --replicas 3 \
  nginx

# Connect service to additional network:
docker service update --network-add my-other-network web-server

# Remove service from network:
docker service update --network-rm my-network web-server
```

### Load Balancing

```bash
# Ingress load balancing (default):
docker service create \
  --name web-server \
  --publish published=8080,target=80,mode=ingress \
  --replicas 3 \
  nginx

# Host mode publishing:
docker service create \
  --name web-server \
  --publish published=8080,target=80,mode=host \
  --replicas 3 \
  nginx
```

### Service Discovery

```bash
# Services can discover each other by name:
# In docker-compose.yml:
version: '3.8'
services:
  web:
    image: nginx
    # Can connect to 'api' service by hostname 'api'
  api:
    image: myapp:latest
    # Can connect to 'db' service by hostname 'db'
  db:
    image: postgres
```

## 9. Storage and Volumes {#storage}

### Volume Types

```bash
# Named volumes:
docker volume create my-volume

# Create service with named volume:
docker service create \
  --name database \
  --mount source=my-volume,target=/var/lib/mysql \
  mysql:8.0

# Bind mounts (not recommended for production):
docker service create \
  --name web-server \
  --mount type=bind,source=/host/path,target=/container/path \
  nginx

# tmpfs mounts (temporary filesystem):
docker service create \
  --name cache \
  --mount type=tmpfs,target=/tmp,tmpfs-size=100m \
  redis
```

### NFS Volumes

```bash
# Create NFS volume:
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.100,rw \
  --opt device=:/path/to/nfs/share \
  nfs-volume

# Use in service:
docker service create \
  --name web-server \
  --mount source=nfs-volume,target=/var/www/html \
  nginx
```

### Volume Plugins

```bash
# Install volume plugin (example with REX-Ray):
docker plugin install rexray/ebs

# Create EBS volume:
docker volume create \
  --driver rexray/ebs \
  --opt size=10 \
  ebs-volume
```

## 10. Security {#security}

### TLS and Certificates

```bash
# Swarm automatically handles TLS:
# - Node-to-node communication is encrypted
# - Certificates are automatically rotated
# - Root CA is created automatically

# View certificate information:
docker swarm ca

# Rotate certificates:
docker swarm ca --rotate

# Update certificate expiry:
docker swarm update --cert-expiry 48h
```

### Secrets Management

```bash
# Create secret from file:
echo "mysecretpassword" | docker secret create db_password -

# Create secret from stdin:
docker secret create ssl_cert ./ssl-certificate.crt

# List secrets:
docker secret ls

# Inspect secret:
docker secret inspect db_password

# Use secret in service:
docker service create \
  --name database \
  --secret db_password \
  --env MYSQL_ROOT_PASSWORD_FILE=/run/secrets/db_password \
  mysql:8.0

# Remove secret:
docker secret rm db_password
```

### Config Management

```bash
# Create config:
docker config create nginx_config ./nginx.conf

# List configs:
docker config ls

# Use config in service:
docker service create \
  --name web-server \
  --config source=nginx_config,target=/etc/nginx/nginx.conf \
  nginx

# Remove config:
docker config rm nginx_config
```

### RBAC and Access Control

```bash
# Manager nodes have full control
# Worker nodes have limited access

# Best practices:
# - Use separate user accounts for different team members
# - Implement proper network segmentation
# - Use secrets for sensitive data
# - Regular security updates
# - Monitor and audit access
```

### Monitoring Stack

```yaml
# monitoring-stack.yml
version: '3.8'

services:
  prometheus:
    image: prom/prometheus
    ports:
      - "9090:9090"
    volumes:
      - prometheus_data:/prometheus
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--web.console.libraries=/etc/prometheus/console_libraries'
      - '--web.console.templates=/etc/prometheus/consoles'
      - '--storage.tsdb.retention.time=200h'
      - '--web.enable-lifecycle'

      replicas: 1
      placement:
        constraints:
          - node.role == manager

  grafana:
    image: grafana/grafana
    ports:
      - "3000:3000"
    volumes:
      - grafana_data:/var/lib/grafana
    environment:
      - GF_SECURITY_ADMIN_USER=admin
      - GF_SECURITY_ADMIN_PASSWORD=admin
    deploy:
      replicas: 1

  node-exporter:
    image: prom/node-exporter
    ports:
      - "9100:9100"
    volumes:
      - /proc:/host/proc:ro
      - /sys:/host/sys:ro
      - /:/rootfs:ro
    command:
      - '--path.procfs=/host/proc'
      - '--path.rootfs=/rootfs'
      - '--path.sysfs=/host/sys'
      - '--collector.filesystem.ignored-mount-points=^/(sys|proc|dev|host|etc)($$|/)'
    deploy:
      mode: global

  cadvisor:
    image: gcr.io/cadvisor/cadvisor
    ports:
      - "8080:8080"
    volumes:
      - /:/rootfs:ro
      - /var/run:/var/run:rw
      - /sys:/sys:ro
      - /var/lib/docker/:/var/lib/docker:ro
    deploy:
      mode: global

volumes:
  prometheus_data:
  grafana_data:
```

## 12. High Availability and Scaling {#ha-scaling}

### Manager Node HA

```bash
# Recommended manager configurations:
# 3 managers: can tolerate 1 failure
# 5 managers: can tolerate 2 failures  
# 7 managers: can tolerate 3 failures

# Add manager nodes:
docker swarm join-token manager

# Best practices:
# - Odd number of managers
# - Distribute across availability zones
# - Use dedicated manager nodes for large clusters
```

### Auto Scaling

```bash
# Manual scaling:
docker service scale web-server=10

# Scaling based on CPU/memory (requires external tools):
# - Docker Swarm doesn't have built-in auto-scaling
# - Use external solutions like Docker Swarm Autoscaler
# - Consider Kubernetes for advanced auto-scaling
```

### Health Checks

```bash
# Service with health check:
docker service create \
  --name web-server \
  --health-cmd "curl -f http://localhost:80 || exit 1" \
  --health-interval 30s \
  --health-timeout 10s \
  --health-retries 3 \
  nginx
```

### Circuit Breaker Pattern

```yaml
# In docker-compose.yml
version: '3.8'

services:
  web:
    image: nginx
    deploy:
      restart_policy:
        condition: on-failure
        delay: 5s
        max_attempts: 3
        window: 120s
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost"]
```

## 13. Production Best Practices {#best-practices}

### Cluster Architecture

```bash
# Production cluster recommendations:
# - Minimum 3 manager nodes
# - Separate manager and worker roles
# - Use labels for node organization
# - Implement proper network segmentation
# - Regular backups of swarm state

# Node labeling strategy:
docker node update --label-add environment=production node1
docker node update --label-add zone=us-east-1a node1
docker node update --label-add type=compute node1
```

### Resource Management

```yaml
# Resource limits in production:
version: '3.8'
services:
  web:
    image: nginx
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M
      placement:
        constraints:
          - node.labels.environment == production
```

### Security Hardening

```bash
# Security checklist:
# 1. Enable Docker Content Trust
export DOCKER_CONTENT_TRUST=1

# 2. Use official images or scan custom images
docker scan myapp:latest

# 3. Run containers as non-root user
# Add to Dockerfile:
# RUN addgroup -g 1001 appgroup && adduser -u 1001 -G appgroup -s /bin/sh -D appuser
# USER appuser

# 4. Use secrets for sensitive data
# 5. Enable audit logging
# 6. Regular security updates
# 7. Network segmentation
# 8. Certificate rotation
```

### CI/CD Integration

```yaml
# GitLab CI example (.gitlab-ci.yml):
deploy_to_swarm:
  stage: deploy
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker stack deploy -c docker-compose.prod.yml --with-registry-auth myapp
  only:
    - master
  environment:
    name: production
```

### Backup Strategy

```bash
# Backup swarm certificates and keys:
sudo tar -czf swarm-backup-$(date +%Y%m%d).tar.gz -C /var/lib/docker/swarm .

# Backup application data:
docker run --rm -v volume_name:/data -v $(pwd):/backup alpine tar czf /backup/backup.tar.gz /data

# Backup database:
docker exec database_container mysqldump -u root -ppassword database > backup.sql
```

## 14. Troubleshooting {#troubleshooting}

### Common Issues and Solutions

```bash
# 1. Node connectivity issues
# Check network connectivity:
ping <NODE-IP>
telnet <NODE-IP> 2377

# Check firewall:
sudo ufw status
sudo iptables -L

# 2. Service deployment failures
# Check service logs:
docker service logs <SERVICE-NAME>

# Check service details:
docker service ps <SERVICE-NAME>

# 3. Resource constraints
# Check node resources:
docker node inspect <NODE-NAME> | grep -A 10 Resources

# Check system resources:
docker system df
docker system events

# 4. Network issues
# List networks:
docker network ls

# Inspect network:
docker network inspect <NETWORK-NAME>

# Test connectivity:
docker run --rm --network <NETWORK-NAME> alpine ping <SERVICE-NAME>
```

### Diagnostic Commands

```bash
# Cluster information:
docker info | grep -A 20 Swarm
docker node ls
docker service ls

# System events:
docker system events --filter type=service
docker system events --filter type=node

# Resource usage:
docker stats
docker system df

# Network troubleshooting:
docker network ls
docker network inspect bridge
docker network inspect <OVERLAY-NETWORK>

# Service troubleshooting:
docker service ps <SERVICE-NAME> --no-trunc
docker service inspect <SERVICE-NAME>
docker service logs <SERVICE-NAME>
```

### Recovery Procedures

```bash
# Recover from manager node failure:
# If you lose quorum:
docker swarm init --force-new-cluster --advertise-addr <MANAGER-IP>

# Restore from backup:
sudo systemctl stop docker
sudo rm -rf /var/lib/docker/swarm
sudo tar -xzf swarm-backup.tar.gz -C /var/lib/docker/swarm
sudo systemctl start docker
```

## 15. Migration Strategies {#migration}

### Migrating to Docker Swarm

```bash
# From standalone Docker:
# 1. Plan the migration
# 2. Convert docker run commands to docker service create
# 3. Convert docker-compose.yml to stack format
# 4. Test in staging environment
# 5. Execute migration during maintenance window

# Example migration:
# Old: docker run -d -p 80:80 nginx
# New: docker service create --name web --publish 80:80 nginx
```

### Zero-Downtime Deployments

```yaml
# Rolling update configuration:
version: '3.8'
services:
  web:
    image: nginx:alpine
    deploy:
      replicas: 4
      update_config:
        parallelism: 1
        delay: 30s
        failure_action: rollback
        monitor: 60s
        order: start-first
      rollback_config:
        parallelism: 1
        delay: 0s
        failure_action: pause
        monitor: 60s
        order: stop-first
```

### Blue-Green Deployments

```bash
# Deploy green environment:
docker stack deploy -c docker-compose.green.yml myapp-green

# Test green environment
# Switch traffic to green
docker service update --publish-rm 80:80 myapp-blue_web
docker service update --publish-add 80:80 myapp-green_web

# Remove blue environment:
docker stack rm myapp-blue
```

## Conclusion

Docker Swarm provides a robust platform for container orchestration with a relatively simple learning curve. This guide covers the essential aspects needed for production deployments, from basic setup to advanced operational practices.

### Key Takeaways:
1. **Start simple** - Docker Swarm is easier to learn than Kubernetes
2. **Plan for HA** - Use odd number of managers across availability zones
3. **Security first** - Use secrets, configs, and proper network segmentation
4. **Monitor everything** - Implement comprehensive monitoring and logging
5. **Practice recovery** - Regular backups and disaster recovery testing
6. **Automate deployments** - Integrate with CI/CD pipelines
7. **Scale thoughtfully** - Consider resource limits and placement constraints

### Next Steps:
- Set up a test environment
- Practice with the provided examples
- Implement monitoring stack
- Plan production migration strategy
- Consider migration to Kubernetes for more advanced orchestration needs

Remember: Docker Swarm is excellent for teams wanting Docker-native orchestration with minimal complexity, but evaluate Kubernetes for more advanced features like custom resources, operators, and extensive third-party ecosystem.
